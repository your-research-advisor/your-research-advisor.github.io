TL;DR

Intel CPUs run on a hidden microcode layer that translates x86 instructions into internal micro-operations (µops). Researchers at Positive Technologies discovered that this layer can be read, and in some Atom processors, unlocked and patched — exposing a range of undocumented instructions, access to internal CPU state, and entirely new attack surfaces for platform security.

01 What is Microcode — and Why Should You Care?

Every modern Intel processor contains a firmware layer invisible to software: microcode. It sits between the x86 instruction set you write and the actual transistor-level execution hardware. When your CPU fetches an ADD or MOV instruction, microcode translates it into a sequence of simpler, internal micro-operations (µops) that the execution engine actually runs.

This layer is entirely opaque. Intel ships CPUs with microcode burned into a ROM, with an optional RAM overlay that can be patched via OS or BIOS updates. The format, encoding, and semantics of these µops are never published publicly.

Why Microcode Matters for Security

Microcode runs below the OS, hypervisor, and even firmware. A vulnerability or backdoor at this level is essentially invisible to any standard security tool. The researcher's phrase: "access to CMS (internal microcode system) may not be achievable via documented MsrOps, but once accessed — becomes a full processor."

The Instruction Documentation Problem

Intel's architecture manuals describe the x86 ISA. But processors ship with undocumented instructions — opcodes that exist in silicon but appear in no manual. Positive Technologies researchers found these can be activated via specific WRMSR sequences, and that Atom-family processors are particularly susceptible to microcode unlocking.

The 15-bit field in the µop encodes specific execution flow for microcode steps, and a separate 4-bit field embeds match/patch register indices. Two synchronisation fields together reflect the CPU's two-step synchronisation across microcode and hardware pipeline stages.

The opcode field specifies the format of the encoded µop. Three separate "opcode" nodes in a sequence word implement what researchers call µop1, µop2, µop3 — the three-step decode of each machine instruction reaching the complex decoder. This feeds into the 5-way decode pipeline, where simple decoders handle 1–4 µop instructions and the complex decoder covers those requiring 4 or more µops.

04 The LDAT: Reading the Microcode ROM

The LDAT (Local Data Access Tool) is an internal CPU primitive that allows access to internal arrays using specific MSR sequences. Researchers used LDAT to:

  • Read microcode ROM contents directly from the CPU
  • Access the MS-ROM (Microcode Sequencer ROM) entries
  • Enumerate internal CPU arrays (ARRAY0 through ARRAY4)
  • Probe internal state like JTAG ClockPassword registers

Access to LDAT normally requires privileged Ring 0 access on MicroROM (read-only) and Ring -2 for write access. The LDAT opcode formatter encodes fields including: hold_name, p-code format, and opcode size. Several fields directly encode dynamic memory addresses for op-standby.

Attack Relevance

If an attacker achieves Ring -2 (System Management Mode) or exploits an ME/CSME vulnerability, LDAT becomes a powerful tool for reading and potentially modifying live microcode execution — below the visibility of any OS-level security.

The primary analysis technique employed was Intel ME (Management Engine) waypoint static analysis — using the ME firmware as a vector to understand the µ-code layer it initialises. The key insight: the ME's µ-code layer operates at hardware-level instruction granularity, implementing high-level machine code instructions that then implement further microcode.

Tools Used by Researchers

  • Intel VISA / Rabbit Holes — internal silicon bring-up and validation infrastructure
  • Intel Management Engine (JTAG) — access to internal processor state via JTAG interface
  • Intel DCI (Direct Connect Interface) — debugging interface bypassing OS, used for JTAG over USB
  • XuCode — Intel's internal x86-to-microcode compiler / toolchain for complex instruction implementations

06 The Microcode Decoder Architecture

The researchers documented a detailed snapshot of the Intel CPU microcode decoder. The pipeline is:

Microcode Sequencer → 5-Way Decode Pipeline

Microcode Sequencer ROM (MS-ROM) feeds into a 5-way decode stage: one Complex Decoder handling 1–4+ µop instructions, and four Simple Decoders each handling 1-µop instructions. The decoded µops feed into the execution engine. Instruction shape (number of µops) determines scheduling — simple decoders maximise throughput while the complex decoder handles legacy or compound instructions.

The Microcode Decoder itself is implemented as a snapshot from the CPU core structure involving:

  • MicroaddrROM entry point — address lookup for each x86 opcode
  • Comparator — matches incoming decoded opcodes to microcode entries
  • Microcode Sequencer — sequences multi-µop translations
  • NopAddr Register — handles NOP and no-operation optimisation paths
  • MatchPatch Registers — hardware registers that hold patch match criteria for runtime microcode patching

The patch mechanism implements two addresses: 1st address → intercept execution of µprocedure, 2nd address → add path (inside RAM). This is essentially a CPU-level hook — analogous to function hooking but at the microcode execution layer.

07 Microcode Update: The RED-UNLOCK Vector

Intel ships regular microcode updates that patch CPU errata and vulnerabilities (e.g. Spectre, Meltdown mitigations). These updates are:

  • Loaded into Microcode RAM by OS/BIOS at boot
  • Encrypted and signed using an unknown algorithm
  • Distributed via Windows Update or intel-microcode Linux packages
Microcode Update — Encryption Detail

Routine updates are encrypted using CFFT65 (a proprietary transform). The encryption uses an unknown algorithm described as "well-formed" but distinct from documented RSA-2048. The update is signed using a proprietary RSA-2048 signature. The algorithm is documented internally as "Algorithm" but not matched to any public standard.

Critically, for the Intel Atom (Goldmont) family, researchers discovered a RED-UNLOCK capability: by writing a specific MSR sequence, the microcode update verification can be bypassed, allowing arbitrary unsigned microcode to be loaded into Microcode RAM.

The VOID ID for the Atom unlock: 0x20003 ... 0x8000000A1 — with patch payload toBD7Array[2..4] encoding the microcode patch content.

CVE Reference

This research is related to CVE-2019-0043 and the broader Atom Goldmont microcode unlock disclosed by Positive Technologies. It enables arbitrary microcode execution on affected CPUs without Intel signing validation.

08 JTAG & the ClockPassword: Hardware Root Access

JTAG (Joint Test Action Group) is a hardware debug interface present on Intel platforms. The researchers explored JTAG internals, finding:

  • JTAG ClockPassword — a 79-bit value (no brute-force feasible) that gates hardware debug access
  • Unlocking JTAG on Atom-family CPUs means access to all CPU internal state, including the microcode execution flow
  • The unlock path: chip red pill / brute-strip (hardware fuzzing of JTAG password) — feasible only with physical access
  • Once unlocked: access to JTAG System State, currently only on the Device debug path
  • Intel DCI enables JTAG over USB debugging — originally for OEM bring-up, but exploitable if not disabled
Platform Security Implication

On production systems, JTAG should be fused off. However, researchers found some retail and near-retail Atom devices with JTAG still accessible, enabling full hardware-level debug and microcode introspection — "how can internal debug (JTAG) debug OS firmware? DCI — debug OS firmware UEFI."

09 Access to CMS: The Undocumented Instructions

The Positive Technologies research centred on accessing the CMS (CPU Microcode System) — the internal execution environment of the microcode layer itself. Key findings:

  • CMS may not be accessible via documented MsrOps (Model Specific Register operations)
  • But once accessed, it provides full microcode-level control — including read/write to microcode RAM, patch registers, and internal CPU state
  • The Undocumented CAO (Chip Architecture Operation) instructions can read/write this state
  • Delivery mechanism: writing specific undocumented MSR sequences from Ring 0 on unlocked Atom processors
Still Open Questions

The researchers flagged that the sub-$0.09 bug bounty for critical microcode vulnerabilities reflects Intel's risk modelling — physical access is required for most of these. But with DCI enabled over USB, some of these attack paths become remote enough to warrant serious concern at the platform level.

— Conclusion

The Positive Technologies research on Intel microcode is one of the most technically detailed public explorations of a CPU's hidden execution layer. The findings — from the 72-bit µop format to the RED-UNLOCK Atom vulnerability — reveal a world of undocumented complexity that exists in every Intel processor.

The key takeaways: microcode is not just a performance detail, it is a security boundary. When that boundary is crossed — whether via JTAG, DCI, or undocumented MSR sequences — the attacker has access to the most privileged execution context possible: below the OS, below the firmware, below every security control we rely on.

Understanding this layer is no longer optional for serious platform security engineers.
